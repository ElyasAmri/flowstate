One line: we turn a government procedure into a deterministic state machine an agent harness runs. Routine cases flow through, only exceptions reach a human.

The problem

Every case waits on a person.

  • Slow. Routine applications queue behind a human reviewer.
  • Inconsistent. The same case decided two ways on two desks.
  • Expensive. Human time spent on cases that never needed it.
  • Opaque. No replayable, appealable record of why.

Yet…

Most applications are routine. Only a few are genuine exceptions. We pay the full human cost for all of them.

25% of scoring: importance, societal impact, value prop. Gulf/Arabic context: dialect + local procedures.

The insight

Automate the routine deterministically.
Escalate only the exceptions.

Keep the whole thing accountable: replayable and appealable.

This is the thesis of the entire project. Pause here.

The core idea

One rule makes it auditable.

The harness is deterministic and sealed. It never touches the outside world directly.

Everything outside (a citizen's app, a registry, a bureaucrat's desk, even another flow) is reached only across a channel: a named, typed boundary you can see on the canvas.

Every external interaction is named, typed, and replayable.

◀⎯⎯▶ citizen app
▶⎯⎯ eID / registry
◀⎯⎯▶ nested flow
sealed deterministic harness
The boundary rule is what buys auditability. Channels live in a registry, one file each, referenced by id.

Four node kinds

Color encodes what's across the boundary.

ui channel

Channel → a person

Crosses to a human-operated app: citizen, bureaucrat, hotline.

service channel

Channel → a service

Crosses to an internal/external service: eID, registry, payment.

flow channel

Channel → another flow

The far side is a nested flow. This is how flows compose.

agent

Agent

The one non-deterministic step: an AI reasons, classifies, drafts.

action / decision: deterministic internal logic (gray)
Channel color is derived from its binding, not stored on the node. Agent is the only non-deterministic node. Action/decision are pure logic.

generated: .maestro/flows/residence-certificate.yaml


version: 1
initial: intake
nodes:
  intake:
    kind: action
    action: log
    message: Application received
    on:
      - to: validate-id
  classify-proof:
    kind: agent
    agent: arabic-reasoner
    prompt: |
      Classify the proof of address. End with VERDICT: clear|unclear.
    on:
      - when: verdict == "clear"
        to: issue
      - to: bureaucrat
Determinism lives in the graph, exactly as the harness expects. The compiler is real: compile.ts, among 68 passing unit tests.

Fanar & external tools

Where the model does the work.

  • Harness backend: Fanar runs the agent nodes and the flow-mining step.
  • Document parsing: reads uploaded evidence (text, images, transcripts) into the structured verdict a flow branches on.
  • Arabic hotline: dialect-aware speech as a channel into the same flow.
  • eID, registry, payment: wired as service channels.

Why Fanar

Gulf government procedures are Arabic-first and dialect-heavy. The model that reasons over the case must be too, and stay inside the audited boundary.

20% of scoring: meaningful integration, strengths, limits. Be honest about stubbed vs live for the demo.

Evaluation & lessons

Tested live, on real Fanar.

100%
routine vs exception — a deterministic guard, all four models
98%
Arabic structured verdict on Fanar (100% Claude / DeepSeek / GPT-5.5)
56–84%
blind judgement, court reasoning stripped — the hard tail every model fails

Real judgement is hard, so the model never decides alone: the ambiguous tail is exactly what the human gate catches. Self-hosted on RunPod; swapping models is one config line.

15% of scoring. Numbers are live: conformance is the deterministic split; Arabic is reading + structured output (Fanar's strength); the facts-only judgement track strips the court's reasoning so the model decides blind, and everyone drops. Recommendation for Fanar: native tool-calling and calibrated confidence to drive the gate.
