Why aren't agents in government?

It comes down to reliability.

  • We rely on others to do work correctly, and to be accountable when it is wrong.
  • Agents are unreliable on some tasks —
  • and inconsistent: different answers to the same input.

So: automate the routine, escalate the exceptions, stay accountable.

Why aren't LLM agents inside government work? Reliability. We rely on people to do work correctly and be accountable when it's wrong. Agents are both unreliable on some tasks and inconsistent — different answers to the same input. Flowstate's answer: automate the routine deterministically, escalate only the real exceptions to a human, and keep the whole thing accountable.

Flowstate

Input arrives over channels → runs through code, agents, services, and nested flows → pauses at human gates — all in one sealed harness.

Flowstate is an agent-driven workflow editor, inspired by Claude Code and n8n. You author a workflow: input arrives over channels, flows through code, agents, external services, and nested flows, and at key steps it pauses for a human or messages another channel — all in a single deterministic harness. [Talk over the demo: the draft flow mines the routine procedure; the procedure runs and escalates an ambiguous case to a human gate; the update flow proposes a change back.]

Separation of concerns

  • A flow is a deterministic state machine — not one over-loaded agent.
  • A self-improving loop: draft → run → update — mine the procedure, run it, improve it from its own exceptions.

Three procedures, one engine.

Instead of handing one agent every tool and task, Flowstate models separation of concerns: the control flow is a deterministic state machine; agents handle only the judgement steps. It's a closed loop of three flows — one drafts the procedure from historical data, one runs it, one turns accumulated exceptions back into proposed updates. This maps onto the agentic requirements: planning and decomposition, tool orchestration, memory/state, autonomous execution, and human-in-the-loop. We validated it on three different procedures.

Where Fanar fits

  • Agent backend — the nodes that reason, classify, and call tools.
  • Document parsing — text, images, and audio transcripts.
  • Strong at extracting meaning from Arabic context — a fit for the Gulf-government setting.

One OpenAI-compatible config entry — online (Fanar API) ↔ self-hosted, swapped without code changes.

Fanar plays two roles: the agentic LLM behind tool-calling nodes, and a document parser across text, images, and audio. Its Arabic strength is a perfect fit for a Gulf-government setting. And it's reached through one OpenAI-compatible config entry — online via the Fanar API or self-hosted on our own instance — swapped without touching code.

We pulled procedure datasets to test effectiveness: a road-traffic fine event log and an Arabic legal-judgement set. Scoring is done by script against a held-out key — the model never grades itself. Fanar on a single self-hosted instance is on par with frontier models for Arabic extraction and conformance, and slightly under on the hardest reasoning task. Beyond accuracy, the update flow mined five concrete, data-backed improvements to the procedure from the 3% of cases that were appealed.
