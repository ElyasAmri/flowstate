The problem

Why aren't agents running institutions?

  • Unreliable on the tasks we hand them.
  • Inconsistent: same input, two different answers.
  • Unaccountable: no one answerable for the step.
The reliability framing (script): institutions rely on work being done correctly and someone being held accountable. Agents are not only fallible, they are inconsistent on the same input, so you can't drop one into the middle of a procedure.

The root cause

One agent, too many responsibilities.

Most work is a fixed order of steps. Split the concerns; coordinate the agents.

Too many tools, too many tasks, expecting wonders. The thesis: separation of concerns over the agents buys consistent, reliable output. Pause here.

Flowstate

Coordinate the steps, not one big agent.

  • Author a workflow in an editor.
  • Steps: code, agents, services, nested flows.
  • Crossed by typed channels; pause for a human anytime.
  • One sealed harness runs it all.

Flowstate (script): agent-driven, inspired by Claude Code and n8n. An editor authors a workflow; input crosses channels; steps are code, agents, external services, nested flows; a step can pause for a human or message another channel; one harness gives maximum control.

Use of Fanar

Two roles in every flow.

  • Agentic LLM — drives the agent nodes (tool-calling).
  • Document parsing — text, images, audio → a typed verdict.

Why Fanar

Arabic-first procedures. Fanar reads them best.

Script: Fanar's two main roles are agentic tool-calling and document parsing (image / text / audio); its Arabic extraction is the perfect fit. Gulf procedures are Arabic-first and dialect-heavy. Hotline and the eID / registry / payment integrations are in the appendix.

Four node kinds

Color encodes what's across the boundary.

ui channel

Channel → a person

Crosses to a human-operated app: citizen, bureaucrat, hotline.

service channel

Channel → a service

Crosses to an internal/external service: eID, registry, payment.

flow channel

Channel → another flow

The far side is a nested flow. This is how flows compose.

agent

Agent

The one non-deterministic step: an AI reasons, classifies, drafts.

action / decision: deterministic internal logic (gray)
Channel color is derived from its binding, not stored on the node. Agent is the only non-deterministic node. Action/decision are pure logic.

From canvas to runtime

The editor compiles to a runnable flow.

The authored graph compiles to maestro YAML the harness loads and validates before running.

  • guards become when:
  • edge assignments become set:
  • agent / action / human-gate map to harness node kinds

generated: .maestro/flows/residence-certificate.yaml


version: 1
initial: intake
nodes:
  intake:
    kind: action
    action: log
    message: Application received
    on:
      - to: validate-id
  classify-proof:
    kind: agent
    agent: arabic-reasoner
    prompt: |
      Classify the proof of address. End with VERDICT: clear|unclear.
    on:
      - when: verdict == "clear"
        to: issue
      - to: bureaucrat
Determinism lives in the graph, exactly as the harness expects. The compiler is real: compile.ts, among 68 passing unit tests.

Solution architecture

Discovery mines a draft flow from datasets + policy. Three user roles: policy maker authors, bureaucrat signs off, consumer is served. The sealed harness runs the flow; external and core services are reached only across channels (extract, call, persist, log); Fanar is the model backend. The edge labels are the contracts: ingest, discover, write, read, run, interact, intervene.
