Why aren't agents in government?

It comes down to reliability.

  • We rely on others to do work correctly, and to be accountable when it is wrong.
  • Agents are not only unreliable on some tasks,
  • they are also inconsistent: different answers to the same input.
Before we started, we asked ourselves one question: why aren't LLM agents integrated into the work of institutions and government. One technical issue is reliability. Us humans tend to rely on others doing work correctly and be held accountable when done incorrectly. Agents are not only unreliable on certain tasks, they are also inconsistent when repeating the task on the same input.

Too many responsibilities

  • We hand one agent too many tools and too many tasks, and expect wonders.
  • But most work follows a fixed ordering of steps.
  • So: model separation of concerns across agents and coordinate them, for consistent, reliable output.

And this is where Flowstate comes in.

Coming from a software engineering background, this problem is easily attributable to one reason: agents have too many responsibilities. We hand agents too many tools, too many tasks, and expect to do wonders. But the truth is that most work, if not all, follows a certain ordering or steps. So the question is, can we model separation of concerns on the agents and coordinate them to achieve more consistent and potentially reliable output? And this is where Flowstate comes into the picture.

Flowstate

An editor for workflows: input arrives over channels, runs through code, agents, services, and nested flows, pauses at human gates, all in one sealed harness.

Flowstate is an agent-driven program, inspired by claude code and n8n. It has an editor which allows users to create a workflow to handle input through outside sources (channels), funnel input through multiple steps which of code, agents, external service, and nested flows, and at certain steps asks for intervention from a human worker or send messages to other channels. All of this is run in a single harness, which allows for maximum control over each step. [Talk over the demo: the init flow drafts the routine procedure; the procedure runs and escalates an ambiguous case to a human gate; the update flow proposes a change back to it. Where Fanar fits

  • Agentic LLM: the agent nodes that reason, classify, and call tools.
  • Document parsing: text, images, audio (transcripts).
  • Strong at extracting meaning from Arabic context, a perfect fit for the Gulf government setting.
Fanar serves two main roles in this program: agentic LLM for tool calling, and document parsing (image, text, audio etc.) Its capability to better extract info from Arabic context is a perfect suit for this. One config entry; online (Fanar API) or self-hosted on RunPod, swapped without code changes.
